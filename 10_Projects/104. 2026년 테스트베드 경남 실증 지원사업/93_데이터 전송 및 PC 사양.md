머시닝 센터(MCT) 컨트롤러에서 발생한 물리적 신호가 유니티(Unity) 가상 환경에서 실시간 3D 동작으로 시현되기까지의 하드웨어 및 소프트웨어 연결 과정을 단계별로 정리해 드립니다.

### 1\. 데이터 발생 및 추출 단계 (Acquisition)

실제 가공 현장의 MCT 장비 내부에서 발생하는 로우 데이터를 수집하는 단계입니다.

- **하드웨어 (MCT Controller):** 가공 공정 중 FANUC 또는 Mitsubishi 컨트롤러 내부의 레지스터에 주축 RPM, 이송 속도(Feed), 스핀들 부하(Load), 알람 코드 등의 파라미터가 실시간으로 생성됩니다.
- **물리적 연결:** 컨트롤러의 내장 이더넷 포트 또는 고속 시리얼 버스(HSSB)를 통해 외부 게이트웨이나 PC와 연결됩니다.
- **소프트웨어 인터페이스:** 제조사 전용 라이브러리인 **FANUC FOCAS2** 또는 **Mitsubishi NC API** 를 호출하여 폐쇄적인 컨트롤러 메모리 주소(예: acts, actf 등)에서 수치 데이터를 읽어옵니다.

### 2\. 데이터 규격화 및 전송 단계 (Middleware)

수집된 장비 특화 데이터를 범용적인 엔진(Unity)이 이해할 수 있는 표준 규격으로 변환하여 전송하는 단계입니다.

- **하드웨어 (Edge Gateway):** 현장에 설치된 산업용 PC 또는 **Advantech ECU-1251**, **Moxa MGate** 와 같은 엣지 게이트웨이가 이 역할을 수행합니다.
- **미들웨어 소프트웨어:****Kepware KEPServerEX** 나 자체 개발된 연계 미들웨어가 수집된 데이터를 **OPC UA** 또는 **MTConnect** 표준으로 정규화(Normalization)합니다.
- **통신 프로토콜:** 유니티 클라이언트와의 0.5초 이내 실시간 동기화를 위해, 오버헤드가 적고 양방향 스트리밍이 가능한 **WebSocket(웹소켓)** 방식으로 데이터를 패킷화하여 발신합니다.

### 3\. 클라이언트 수신 및 처리 단계 (Processing)

유니티 엔진 내부에서 네트워크 데이터를 받아 그래픽 처리에 적합하도록 가공하는 단계입니다.

- **소프트웨어 (Unity Assets):****NativeWebSocket** 또는 **realvirtual.io Professional** 과 같은 라이브러리를 사용하여 미들웨어로부터 전송된 웹소켓 메시지를 수신합니다.
- **데이터 최적화:** 수신된 데이터의 튀는 현상을 방지하기 위해 **노이즈 필터링** 을 적용하며, 엔진의 프레임 속도와 일치하도록 **데이터 보간(Interpolation)** 처리를 거칩니다.

### 4\. 시각화 및 가상 작동 시현 단계 (Visualization)

최종적으로 수치 데이터를 3D 모델의 움직임으로 변환하는 단계입니다.

- **상태 머신(State Machine):** 컨트롤러의 상태값(가동, 정지, 알람 등)에 따라 유니티 내 가상 모델의 애니메이션 레이어를 전환합니다.
- **동작 매핑(Mapping):** 정규화된 RPM 수치는 3D 모델 주축의 회전 속도 변수로, X/Y/Z 좌표 데이터는 가상 오브젝트의 \*\*Transform(위치값)\*\*으로 실시간 매핑되어 실제 장비와 동일한 궤적으로 작동합니다.
- **시각적 피드백:** 스핀들 부하 데이터가 임계치를 넘으면 셰이더(Shader)를 통해 모델의 색상을 변경하거나 진동 효과를 주어 관리자에게 이상 징후를 시각적으로 알립니다.

### 단계별 상세 연결 과정

1. **신호 발생 (Acquisition)**
	- **MCT Controller:** 공정 중 실시간 데이터(RPM, 피드 등)가 컨트롤러 메모리 레지스터에 기록됩니다.
	- **Interface:** 장비의 **Ethernet** 포트 또는 \*\*HSSB(고속 시스템 버스)\*\*를 통해 외부로 전송 가능한 물리적 통로를 확보합니다.
2. **데이터 추출 및 변환 (Middleware)**
	- **Extraction:****FANUC FOCAS2** 또는 **Mitsubishi NC API** 라이브러리를 호출하여 기계 고유의 로우 데이터를 수집합니다.
	- **Normalization:** 수집된 데이터를 **OPC UA** 또는 **MTConnect** 와 같은 표준 데이터 모델로 정규화합니다.
	- **WebSocket Server:** 유니티와의 초고속 연동을 위해 웹소켓 방식으로 데이터를 스트리밍 서버에 로드합니다.
3. **데이터 수신 및 보정 (Processing)**
	- **Unity Client:****NativeWebSocket** 또는 **realvirtual.io** 에셋을 통해 서버에서 밀어주는(Push) 데이터 패킷을 실시간 수신합니다.
	- **Stabilization:** 데이터 통신 중 발생하는 노이즈를 필터링하고, 실제 기계의 부드러운 움직임을 위해 데이터 사이의 값을 메우는 **보간(Interpolation)** 로직을 실행합니다.
4. **가상 시현 (Visualization)**
	- **State Machine:** 수신된 '가동/정지/알람' 상태값에 따라 3D 모델의 애니메이션 상태를 즉각 전환합니다.
	- **Transform Mapping:** 실시간 좌표 데이터를 유니티 오브젝트의 **Transform(위치/회전)** 값에 매핑하여 가상 장비가 실제와 동일하게 움직이도록 시현합니다.
	- **Shader Feedback:** 과부하나 알람 발생 시 3D 모델의 색상을 적색으로 변경하거나 흔들림 효과를 주어 관리자에게 경고를 시각화합니다.

이 아키텍처는 **0.5초 이내의 실시간 동기화** 를 보장하기 위해 각 단계에서의 오버헤드를 최소화하도록 설계되었습니다.

Robustel EG5100을 MCT와 직접 연결하는 건가?

---

네, **Robustel EG5100은 MCT(머시닝 센터) 컨트롤러와 이더넷 케이블을 통해 물리적으로 직접 연결** 됩니다.

이 연결 과정은 실시간 데이터 수집의 가장 기초적인 단계로, 상세한 연결 방식과 설정 요구사항은 다음과 같습니다.

### 1\. 물리적 연결 방식

- **연결 매체:** 표준 이더넷(Ethernet) 케이블을 사용합니다.
- **연결 지점:** MCT 컨트롤러 내부에 내장된 이더넷 포트(주로 조작반 뒤쪽이나 전기 제어함 내부에 위치)와 Robustel EG5100의 **LAN 포트** 를 직접 연결합니다.
- **포트 구성:** EG5100은 소프트웨어로 설정 가능한 RS-232/RS-485 및 CAN 인터페이스도 지원하지만, FOCAS2와 같은 고속 데이터 추출을 위해서는 주로 **이더넷 포트** 를 사용합니다.

### 2\. 네트워크 및 소프트웨어 설정

- **IP 주소 할당:** 컨트롤러와 게이트웨이의 LAN 포트에 동일한 서브넷상의 고정 IP를 할당해야 합니다(예: CNC는 192.168.1.10, 게이트웨이는 192.168.1.11).
- **프로토콜 활성화:** FANUC 컨트롤러의 경우, 이더넷을 통한 데이터 통신 기능인 **FOCAS 옵션** 이 활성화되어 있어야 하며, FOCAS 포트(기본값 TCP 8193) 설정이 필요합니다.
- **데이터 추출 로직:** EG5100 내부의 **RobustOS Pro(데비안 기반)** 환경에 FOCAS 드라이버 또는 관련 애플리케이션(Edge2Cloud Pro 등)을 실행시켜 컨트롤러 레지스터의 데이터를 읽어옵니다.

### 3\. 보안 권장 사항

- MCT와 게이트웨이 간의 연결은 외부 보안 위협으로부터 보호하기 위해 \*\*독립된 네트워크 세그먼트(Isolated Network)\*\*로 구성하는 것이 권장됩니다.
- EG5100은 방화벽 기능을 갖추고 있어 장비를 회사 전체 네트워크와 분리하면서도 필요한 데이터만 상위 PC(Unity)로 안전하게 전송할 수 있는 보안 거점 역할을 수행합니다.

요약하자면, EG5100은 현장에서 MCT의 '입과 귀' 역할을 수행하기 위해 **장비와 1:1로 직접 연결** 되어 폐쇄적인 기계 신호를 수집하고, 이를 Unity PC가 이해할 수 있는 웹소켓(WebSocket) 데이터로 변환하여 쏴주는 다리 역할을 합니다.

---

만약 특정 pc에서 MCT 파라미터 데이터를 받고 있는 상황이라면 그 데이터를 unity engine pc로 가져가기 위해 어떻게 해애 해?

---

특정 PC(데이터 수집용 PC-A)가 이미 MCT 데이터를 받고 있는 상황에서 이를 \*\*유니티 엔진 PC(시연용 PC-B)\*\*로 실시간 전송하려면, PC-A를 일종의 \*\*'데이터 중계 서버(WebSocket Server)'\*\*로 만들고 PC-B를 \*\*'클라이언트(Client)'\*\*로 설정하여 연결해야 합니다.

가장 권장되는 실시간 전송 방식(0.5초 이내 동기화용)은 다음과 같습니다.

### 1\. 전송 아키텍처 설정

PC-A에서 받은 데이터를 PC-B로 보내기 위해 '발행-구독(Pub/Sub)' 또는 '스트리밍' 구조를 구축합니다.

- **네트워크 구성:** 두 PC를 동일한 작업장 내 \*\*로컬 네트워크(LAN)\*\*에 연결합니다. 병목 현상을 방지하기 위해 유선 기가비트 이더넷 연결을 권장합니다.
- **PC-A (서버 역할):** MCT로부터 받은 로우 데이터를 JSON 규격 등으로 정규화한 후, 웹소켓(WebSocket) 프로토콜을 통해 외부로 쏴주는 서버 프로그램을 실행합니다.
- **PC-B (클라이언트 역할):** 유니티 엔진 내에서 PC-A의 IP 주소와 포트 번호(예: `ws://192.168.1.50:8080`)로 접속하여 데이터를 수신합니다.

### 2\. 하드웨어/소프트웨어별 실행 방법

#### 방법 A: 전문 미들웨어 활용 (안정성 위주)

PC-A에 데이터 허브용 소프트웨어를 설치하는 방식입니다.

1. **PC-A에 Kepware 설치:****Kepware KEPServerEX** 의 IoT Gateway 플러그인을 사용하여 수집된 MCT 파라미터를 **웹소켓(WebSocket)** 또는 **MQTT** 로 변환합니다.
2. **PC-B 유니티 에셋 사용:** 유니티에서 **OPCUA4Unity** ($250) 또는 **realvirtual.io** 를 사용하여 PC-A의 데이터를 실시간 구독(Subscribe)합니다.

#### 방법 B: 자체 통신 브릿지 개발 (비용 절감/최적화 위주)

PC-A에서 돌아가는 가벼운 전송 프로그램을 C#이나 Python으로 작성하는 방식입니다.

1. **PC-A 전송 프로그램:** PC-A가 MCT 데이터를 받는 소스 코드에 **WebSocket Server** 기능을 추가합니다. 데이터가 들어올 때마다 접속된 PC-B로 즉시 푸시(Push)하도록 설계합니다.
2. **PC-B 유니티 통신부:** 유니티의 오픈소스 라이브러리인 **NativeWebSocket** 을 사용하여 비동기로 데이터를 수신합니다.
	- **데이터 예시:**`{"RPM": 1200, "Feed": 350, "Load": 45}` 와 같은 JSON 문자열 형태가 처리하기에 가장 용이합니다.

### 3\. 실시간성 확보를 위한 핵심 체크포인트 (0.5초 이내)

- **방화벽 개방:** PC-A의 윈도우 방화벽 설정에서 통신에 사용할 특정 포트(예: 8080, 1883 등)가 열려 있는지 반드시 확인해야 합니다.
- **고정 IP 설정:** 통신 연결이 끊기지 않도록 PC-A에 고정 IP를 할당하십시오.
- **데이터 샘플링 속도:** PC-A에서 PC-B로 데이터를 전송할 때 0.1초(100ms) 단위 이하로 데이터를 갱신해야 유니티에서 부드러운 3D 시뮬레이션(60 FPS 이상)이 가능합니다.
- **보간 처리(Interpolation):** 네트워크 지연으로 데이터가 불규칙하게 들어올 수 있으므로, 유니티 내부에서 이전 값과 현재 값 사이를 부드럽게 이어주는 보간 로직을 적용해야 합니다.

요약하면, **PC-A는 데이터를 쏴주는 '방송국(Server)'**, **유니티 PC-B는 이를 받아 화면에 그리는 '수신기(Client)'** 역할을 하도록 소프트웨어 통신 채널을 구성하면 됩니다.

Pc-b를 사무실이나 특정 원격 위치에 둬야한다면?

---

유니티 엔진 PC(PC-B)를 사무실이나 원격지에 두어야 할 경우, 가장 큰 기술적 과제는 **보안을 유지하면서도 0.5초 이내의 실시간성을 확보하는 것** 입니다. 이를 위해 공장 현장(OT)과 사무실(IT) 네트워크를 연결하는 3가지 핵심 솔루션을 제안해 드립니다.

### 1\. 산업용 전용 VPN 게이트웨이 활용 (가장 권장)

앞서 언급한 **Robustel EG5100** 이나 **HMS Ewon** 과 같은 장비는 자체적인 클라우드 기반 VPN 서비스를 제공합니다.

- **연결 방식:**
	1. **공장 현장:** Robustel 게이트웨이를 MCT와 연결하고 공장 인터넷망에 연결합니다.
	2. **사무실 PC-B:** 게이트웨이 전용 VPN 클라이언트(예: RobustVPN 또는 Talk2M)를 설치합니다.
	3. **가상 네트워크:** 사무실 PC가 공장 현장의 로컬 네트워크(LAN)에 직접 연결된 것처럼 인식됩니다.
- **장점:** 고가의 고정 IP 설정 없이도 방화벽을 통과할 수 있으며, 데이터가 전구간 암호화되어 보안이 매우 강력합니다.
- **지연 시간:** 전용 터널을 사용하므로 일반 인터넷망보다 안정적이며, 0.5초 이내의 동기화가 가능합니다.

### 2\. 클라우드 중계 서버 (Cloud Relay) 방식

현장의 데이터를 외부 클라우드(AWS, Azure 등)에 먼저 쏘고, 사무실에서 이를 받아가는 방식입니다.

- **연결 방식:**
	1. **현장 게이트웨이:** 수집된 데이터를 **MQTT** 또는 **WebSocket** 프로토콜을 통해 클라우드 브로커로 전송합니다.
	2. **클라우드 서버:** 데이터를 수신하여 대기 중인 클라이언트(사무실 PC-B)에게 즉시 푸시(Push)합니다.
	3. **사무실 PC-B:** 유니티 내에서 클라우드 주소에 접속하여 데이터를 실시간 스트리밍 받습니다.
- **장점:** 여러 장소(사무실, 관제 센터, 모바일 등)에서 동시에 모니터링하기에 가장 유리한 구조입니다.
- **단점:** 클라우드를 한 번 거치므로 네트워크 상태에 따라 미세한 지연(Latency)이 추가될 수 있습니다.

### 3\. 클라우드 렌더링 스트리밍 (Vagon Streams 등)

사무실 PC의 사양이 낮을 때 유용한 방식으로, 유니티 실행 자체를 고성능 클라우드 서버에서 수행합니다.

- **연결 방식:** 유니티 디지털 트윈 뷰어를 **Vagon Streams** 와 같은 GPU 클라우드 플랫폼에 올립니다.
- **사용자 경험:** 사무실 사용자는 일반 PC나 태블릿의 웹브라우저를 통해 고화질 3D 화면을 스트리밍 받습니다. 유튜브를 보듯 조작하지만, 실제 연산은 클라우드에서 일어납니다.
- **장점:** 사무실 PC 사양에 구애받지 않고 어디서나 고화질 디지털 트윈을 시현할 수 있습니다.

### 1\. 하드웨어 구성 및 연결도

원격지에 서버를 두더라도 현장 장비(MCT)의 신호를 인터넷망으로 올려줄 \*\*'브릿지 장치'\*\*가 현장에 반드시 하나 필요합니다.

- **현장 (Factory):**
	- **MCT 장비:** 컨트롤러(FANUC/Mitsubishi)의 이더넷 포트 활용.
	- **브릿지(Robustel EG5100 등):** MCT와 1:1로 연결되어 데이터를 외부 VPN 터널로 전송하는 통로 역할을 수행합니다.
- **원격지 (Office - PC-B):**
	- **하이엔드 PC:** RTX 5090, Intel Ultra 9 285K 사양의 워크스테이션.
	- **역할:** VPN을 통해 현장 MCT에 가상으로 직접 접속하여 데이터를 수집하고, 그 데이터로 유니티 엔진을 구동합니다.

### 2\. 단계별 구축 방법

#### 1단계: VPN 터널 구축 (네트워크 통합)

원격 PC-B가 현장의 MCT(예: IP 192.168.1.10)를 마치 옆에 있는 것처럼 인식하게 만드는 과정입니다.

- **방법:** Robustel EG5100의 **RobustVPN** 서비스를 활성화합니다.
- **결과:** PC-B에 전용 VPN 클라이언트를 설치하고 접속하면, PC-B는 현장 장비와 동일한 서브넷(192.168.1.x)에 논리적으로 소속됩니다.

#### 2단계: 원격 데이터 수집 서버 설정 (PC-B 내부)

PC-B 내부에 MCT 데이터를 읽어올 미들웨어를 설치합니다.

- **소프트웨어:****Kepware KEPServerEX** 또는 **FANUC FOCAS2 라이브러리** 가 포함된 자체 C# 프로그램을 설치합니다.
- **설정:** 미들웨어의 접속 대상 IP를 현장 MCT의 고정 IP(192.168.1.10)로 지정합니다. VPN 터널을 통해 PC-B가 직접 데이터를 뽑아옵니다.

#### 3단계: 유니티 엔진 연동 및 시현 (PC-B 내부)

수집된 데이터를 동일 PC 내의 유니티 엔진으로 전달합니다.

- **내부 통신:** PC-B 내에서 돌아가는 미들웨어가 웹소켓(WebSocket) 서버 역할을 하고, 유니티가 로컬 호스트(`ws://127.0.0.1`)로 접속하여 데이터를 가져갑니다.
- **시각화:** 수신된 RPM, 피드, 좌표 데이터를 3D 모델에 실시간 매핑하여 시현합니다.

### 3\. 하이엔드 사양을 활용한 고도화 전략 (0.5초 이내 달성)

PC-B의 강력한 연산 성능을 활용하여 원격지의 한계(지연 시간)를 극복할 수 있습니다.

- **대역폭 최적화:** 현장 브릿지(EG5100)에서 데이터를 전송할 때, 0.1초 단위의 **JSON 데이터 패킷** 으로 압축하여 VPN 부하를 줄입니다.
- **데이터 보간(Interpolation):** 원격망의 미세한 패킷 끊김을 방지하기 위해 유니티 내에서 **선형 보간 알고리즘** 을 적용, 3D 장비의 움직임을 부드럽게 유지합니다.

### 1\. 보안을 위한 망 분리 (OT/IT Isolation)

머시닝 센터(MCT)는 공장의 핵심 자산으로, 보안상 외부 인터넷이나 사무실 네트워크에 직접 노출되는 것은 매우 위험합니다.

- **보안 장벽:** EG5100은 MCT가 속한 \*\*폐쇄적인 장비망(OT)\*\*과 유니티 PC가 있는 **사무실/인터넷망(IT)** 사이에서 강력한 방화벽 역할을 수행합니다.
- **위험 차단:** 일반 PC(PC-A)를 직접 외부망에 연결할 경우, 해당 PC를 통해 공장 전체 장비로 악성코드가 확산될 위험이 있으나 게이트웨이는 필요한 데이터 패킷만 선별적으로 전달합니다.

### 2\. 원격 접속의 편의성 (RobustVPN/RCMS)

사무실이나 원격지의 PC-B가 현장의 데이터를 가져오려면 현장 네트워크의 방화벽을 뚫어야 하는데, 이를 위해 고가의 고정 IP를 할당받거나 복잡한 네트워크 설정을 하는 것은 실무적으로 매우 어렵습니다.

- **원클릭 가상 터널:** EG5100은 자체 제공하는 \*\*RobustVPN(RCMS 플랫폼)\*\*을 통해 복잡한 설정 없이도 사무실 PC-B와 현장 장비를 하나의 가상 네트워크(L2/L3 Bridge)로 묶어줍니다.
- **유연한 연결:** 이를 통해 사무실에 있는 개발자가 마치 현장에서 MCT 옆에 앉아 있는 것처럼 장비에 직접 접속하여 파라미터를 실시간으로 모니터링하고 수정할 수 있습니다.

### 3\. 0.5초 이내의 실시간성 및 안정성 확보

본 사업의 핵심 성능 지표인 \*\*'0.5초 이내 동기화'\*\*와 \*\*'99.9% 안정성'\*\*을 달성하기 위해서는 데이터 수집 장치가 가벼워야 합니다.

- **전용 엣지 컴퓨팅:** 일반 윈도우 PC(PC-A)는 OS 업데이트, 백그라운드 프로세스 등으로 인해 예기치 않은 데이터 전송 지연이 발생할 수 있습니다. 반면 EG5100은 산업용 리눅스(Debian) 기반의 전용 기기로서, 오직 데이터 수집 및 전송 작업(Docker 컨테이너 등)에만 모든 자원을 집중합니다.
- **자동 복구:** 시스템 오류 발생 시 자동으로 이전 상태로 되돌리는 **Auto Failback** 기능을 갖추고 있어, 현장 운영자가 없어도 24시간 안정적인 데이터 스트리밍을 보장합니다.

### 4\. 가혹한 현장 환경 내구성

MCT 가공 현장은 전자기 노이즈, 고온, 진동이 빈번하게 발생합니다.

- **산업용 설계:** EG5100은 팬리스(Fanless) 구조와 금속 하우징을 채택하여 **\-40~70°C** 의 극한 환경에서도 문제없이 작동하며, 윈도우 PC보다 전자기 노이즈에 훨씬 강합니다.

**결론적으로,** PC-A가 단순히 장비 옆에서 데이터를 보고 있는 용도라면, **Robustel EG5100은 그 데이터를 '안전하고 빠르게' 가공하여 멀리 떨어진 PC-B(유니티)까지 배달해주는 '전문 보안 집배원'** 역할을 수행하기 위해 필수적입니다.

---

문의하신 \*\*'산업용 이더넷 스위치를 통한 물리적 연결 방식'\*\*은 현장의 장비와 PC, 그리고 외부를 잇는 게이트웨이를 하나의 네트워크망(LAN)으로 묶어 데이터가 막힘없이 흐르게 하는 핵심 인프라 구성입니다. 상세한 설명은 다음과 같습니다.

### 1\. 산업용 이더넷 스위치(Switching Hub)의 역할

일반적인 허브와 달리 **산업용 스위치** 는 공장 현장의 강력한 전자기 노이즈(EMI)와 진동, 고온 환경에서도 안정적인 데이터 전송을 보장하는 네트워킹 장비입니다.

- **중앙 교차로 역할:** MCT, PC-A, Robustel EG5100에서 나오는 모든 이더넷 케이블이 이 스위치에 집결됩니다. 스위치는 장치 간의 데이터 패킷을 정확한 목적지로 전달하는 '교차로' 역할을 수행합니다.
- **기가비트 대역폭 확보:** 0.5초 이내의 실시간성을 위해 1Gbps급 속도를 지원하는 스위치를 사용하여 내부 데이터 병목 현상을 방지합니다.

### 2\. 3대 핵심 장치의 연결 구조

스위치를 중심으로 세 장치가 다음과 같이 물리적으로 맞물리게 됩니다.

- **MCT 컨트롤러 스위치:** 장비 내부의 이더넷 포트(Fanuc의 경우 조작반 뒤 또는 전기함 내 포트)에서 나온 케이블을 스위치에 연결합니다.
- **PC-A(현장 수집 PC) 스위치:** PC-A의 LAN 포트를 스위치에 연결합니다. PC-A는 이 연결을 통해 MCT의 메모리 레지스터(FOCAS2 등)에 접근하여 데이터를 긁어옵니다.
- **Robustel EG5100 스위치:** EG5100의 **LAN1 포트** 를 스위치에 연결합니다. 이를 통해 EG5100은 동일한 망에 있는 PC-A나 MCT와 통신할 수 있는 물리적 상태가 됩니다.

### 3\. 논리적 연결 (IP 및 서브넷 설정)

케이블만 꽂는다고 통신이 되는 것은 아니며, 세 장치가 서로를 인식할 수 있도록 \*\*'주소 체계'\*\*를 통일해야 합니다.

- **동일 서브넷 설정:** 세 장치 모두 동일한 네트워크 대역(예: 192.168.1.xxx) 내에 있어야 합니다.
	- *MCT IP:* 192.168.1.10
	- *PC-A IP:* 192.168.1.11
	- *EG5100 LAN IP:* 192.168.1.12
- **고정 IP 할당:** 유동 IP(DHCP)를 쓰면 주소가 바뀔 때마다 연결이 끊기므로, 모든 장치에 반드시 **수동으로 고정 IP를 할당** 하여 통신 안정성을 확보해야 합니다.

### 4\. 실제 데이터의 흐름 (Data Path)

이 물리적 구성을 통해 데이터는 다음과 같은 경로로 이동합니다.

1. **로컬 수집:** PC-A가 스위치를 거쳐 MCT의 가공 데이터를 0.1초 단위로 가져옵니다.
2. **데이터 중계:** PC-A 내부에 설치된 서버(WebSocket Server)가 수집된 데이터를 로컬망(LAN)에 뿌립니다.
3. **외부 전송:** 같은 스위치에 물려있는 **Robustel EG5100** 이 PC-A가 뿌린 데이터를 가로채어, 자신이 만든 '보안 터널(VPN)'을 타고 사무실의 PC-B로 전송합니다.

### 5\. 구성의 장점

- **유연성:** 향후 MCT가 추가되더라도 스위치의 빈 포트에 꽂기만 하면 즉시 확장 가능합니다.
- **보안성:** EG5100이 외부망(Internet)과 연결되는 통로를 독점하므로, MCT와 PC-A는 외부 해킹 위협으로부터 격리된 안전한 내부망 환경에서 작동할 수 있습니다.

결론적으로, \*\*산업용 이더넷 스위치는 현장의 모든 디지털 자산을 하나로 묶는 '신경망'\*\*이며, Robustel EG5100은 이 신경망의 입구에서 외부 세계(PC-B)와 안전하게 데이터를 주고받는 **'보안 대문'** 역할을 수행하게 됩니다.
