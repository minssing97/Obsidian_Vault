##### 본 과제(디지털트윈) 성능 목표(7개)

○ 성능 목표는 기능 목표가 아닌 검증 가능한 정량 지표로 정의하며, 목표치와 최소허용치를 병기하여 실증 신뢰도를 확보함

세부목표	목표 성능지표(목표 / 최소허용)	목표측정방법
End-to-end 지연시간	P95 0.5초 이내 / P95 1.0초 이내	측정구간: Controller 수집시각(t_ctrl) → Unity 화면 반영시각(t_unity). 측정도구: 양단 타임스탬프 로그. 산식: Latency = t_unity − t_ctrl, 전체 샘플의 P95 산출
데이터 누락률	0.1% 이하 / 0.5% 이하	측정구간: Collector 전송 카운트 vs Unity 수신 카운트. 산식: (누락건수 / 전체건수) × 100. 시퀀스 ID 기반 대조 로그 분석
재연결 복구시간	10초 이내 / 30초 이내	측정도구: 네트워크 장애 주입 스크립트(LAN 케이블 분리/방화벽 차단). 산식: t_recover − t_drop. 3회 이상 반복 측정 평균
상태 전이 일치율	95% 이상 / 90% 이상	Controller 상태(Run/Idle/Alarm)를 정답(Ground Truth)으로 설정. Unity 상태머신 전이 이벤트와 시간 정렬 후 비교. 산식: (일치건수 / 전체전이건수) × 100
Unity 매핑 정확도(파라미터-동작)	95% 이상 / 90% 이상	RPM/Feed/Load 등 핵심 파라미터의 Controller 값과 Unity 반영값을 로그+화면 녹화로 교차검증. 산식: (정상매핑건수 / 전체매핑이벤트) × 100
연속 운전 안정성	8시간 무중단 / 4시간 무중단	측정도구: Unity 런타임 리소스 모니터링(CPU/RAM/VRAM/FPS). 8시간 연속 가동 중 시스템 중단·에러 발생 유무 기록. 증빙: 연속운전 시험성적서
로그/이력 저장 완전성	99% 이상 / 95% 이상	발생 이벤트(알람·상태전이·파라미터 변경) 총 건수 vs DB 저장 건수 비교. 산식: (저장건수 / 발생건수) × 100. 무결성 점검 리포트

###### 시스템 구성도(데이터 흐름)

○ 시스템은 Controller → Collector → Middleware → Unity의 4계층 데이터 흐름 구조로 설계함
○ 구성요소, 통신 방식, 데이터 유형, 품질 기능을 명확히 정의함

```
[MCT Controller]          [Collector(PC-A)]         [Middleware]              [Unity Engine(PC-B)]
  FANUC/Mitsubishi          현장 수집 PC              데이터 규격화/품질          3D 디지털트윈 시현
  ┌──────────┐            ┌──────────────┐         ┌──────────────┐         ┌──────────────┐
  │ 파라미터  │  Ethernet  │ FOCAS2/NC API│ WebSocket│ 정규화/버퍼  │ WebSocket│ 상태머신     │
  │ RPM/Feed │──────────→│ 데이터 추출  │────────→│ 재전송/재연결│────────→│ 동작 매핑    │
  │ Load/Alarm│  LAN      │ 타임스탬프   │  JSON   │ 시퀀스ID검증 │  JSON   │ 알람/UI      │
  │ X/Y/Z    │            │ 시퀀스ID부여 │         │ 로그 저장    │         │ 로그 기록    │
  └──────────┘            └──────────────┘         └──────────────┘         └──────────────┘
```

○ 데이터 흐름 단계별 상세

단계	구성요소	통신 방식	데이터 유형	품질 기능
1단계: 데이터 발생/추출	MCT Controller(FANUC/Mitsubishi)	Ethernet(LAN, TCP 8193)	파라미터(RPM, Feed, Load, X/Y/Z), 알람 코드, 운전상태	FOCAS2/NC API 호출, 100ms 단위 폴링
2단계: 수집/전처리	Collector(현장 PC-A 또는 Edge Gateway)	FOCAS2/EZSocket → WebSocket Server	JSON 패킷(태그명, 값, 타임스탬프, 시퀀스ID)	타임스탬프 부여, 시퀀스ID 생성, 노이즈 필터링
3단계: 규격화/중계	Middleware(PC-A 내장 또는 별도 모듈)	WebSocket(ws://IP:8080)	정규화 데이터(데이터사전 기반 태그·단위 통일)	버퍼링, 재전송(누락 시), 자동 재연결(keep-alive), 로그 DB 저장
4단계: 시각화/시현	Unity Engine(PC-B)	WebSocket Client(NativeWebSocket)	매핑 데이터(파라미터→Transform/Animation/UI)	데이터 보간(Interpolation), 상태머신 전이, 알람 셰이더 피드백

###### 하드웨어/소프트웨어 구성(93_데이터 전송 및 PC 사양 기반 최적 선정)

**네트워크 연결 구조**

○ 현장→원격: EG5100이 RobustVPN 암호화 터널을 생성하여 원격 PC-B를 현장 LAN에 논리적 편입
○ 원격(Office): PC-B가 VPN 접속 후 PC-A의 WebSocket Server(ws://192.168.1.11:8080)에 직접 연결하여 실시간 데이터 수신

**수행내용 2: 매핑 로직 설계**

○ Controller 파라미터와 Unity 3D 오브젝트 간의 매핑 규칙을 정의함
- RPM → Unity 주축 오브젝트의 Rotation Speed(Angular Velocity) 파라미터에 선형 매핑
- Feed → 테이블/축 오브젝트의 Transform 이동 속도에 비례 매핑
- Load → 부하 게이지 UI 값 + 임계치(80%) 초과 시 셰이더 색상 변경(녹색→황색→적색)
- X/Y/Z → 3D 오브젝트 Transform.position에 좌표계 변환 후 직접 바인딩
- 운전상태(Run/Idle/Alarm) → Unity Animator의 State Machine Trigger로 전이
○ 데이터 보간(Interpolation): 네트워크 지연/불규칙 수신 시 이전 값과 현재 값 사이를 선형 보간하여 60FPS 이상의 부드러운 동작 재현

**수행내용 3: 상태머신 구현**

○ Unity Mecanim 기반 상태머신을 MCT 운전상태에 특화하여 구현함
- 상태 정의: Idle(정지) / Run(가동) / Alarm(알람) / ToolChange(공구교환, 선택)
- 전이 조건: Controller의 MCT.Status.Mode 값 변경 시 트리거 발동
- 시각적 피드백: Run 시 주축 회전+축 이동 애니메이션, Alarm 시 적색 점멸+알람 팝업, Idle 시 정지 상태
- 알람 이력: 알람 발생/해제 시점, 코드, 지속시간을 DB에 자동 저장
